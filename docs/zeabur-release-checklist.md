# Zeabur 发布前检查单

适用场景：

- 将 `DPTEST` 发布到 Zeabur
- 目标域名为 `https://deltapex.top`
- 本次重点验证 lead MVP 和客户侧/销售侧边界是否正确

---

## 一、发布前准备

### 1. 仓库状态

- 当前分支包含本次准备上线的改动
- `npm run check` 通过
- 如涉及结果页或报告页，确认客户侧没有新增内部字段透传

### 2. Zeabur 服务确认

- 已存在或已创建 `web` 服务
- 已存在或已创建 PostgreSQL 服务
- `web` 服务使用仓库中的 `Dockerfile`
- `web` 服务监听端口为 `5000`

### 3. 环境变量

必须存在：

```env
DATABASE_URL=postgresql://<user>:<password>@<host>:5432/<db>
SESSION_SECRET=<openssl rand -hex 32 生成>
ADMIN_PASSWORD=<后台管理密码>
BASE_URL=https://deltapex.top
VECTORENGINE_API_KEY=<实际值>
```

按需存在：

```env
EXTERNAL_API_KEY=
WECHAT_WEBHOOK_URL=
SERA_PASSWORD=
DEVEN_PASSWORD=
ANNA_PASSWORD=
```

### 4. 域名

- `deltapex.top` 已绑定到 Zeabur 对应服务
- DNS 解析已生效
- Zeabur 证书状态正常

### 5. 数据库与启动逻辑

- 数据库连接串指向当前 Zeabur PostgreSQL
- 知道 `docker-entrypoint.sh` 会在启动时做：
  - 数据库可用性检查
  - 表结构补齐
  - 默认账号 seed

---

## 二、发布操作

1. 在 Zeabur 中选择当前 `web` 服务
2. 确认最新 commit 或最新仓库代码已同步
3. 触发重新部署
4. 观察构建日志
5. 观察运行日志

运行日志中至少确认这些事实：

- 容器启动成功
- 数据库连接成功
- 表结构补齐完成
- 应用开始监听 `5000`

如果这里失败，先不要继续做页面验收。

---

## 三、发布后最小冒烟

### 客户侧

- `https://deltapex.top/` 可打开
- 测评流程可进入结果页
- 如有分享链接，`/report/:token` 可打开

### 销售侧

- `/admin/chat` 可登录
- `/lead/mobile` 可访问
- `/lead/ops-import` 可访问
- `/lead/admin` 可访问

### Lead 链路

- 导入预检接口正常
- 导入提交正常
- 销售队列正常
- 标记有效正常
- 标记无效正常

---

## 四、客户侧边界复查

这是本次发布的关键验收项。

### 结果页必须保留

- 交易人格
- 雷达图/能力轮廓
- 六维详细分数
- 核心优势
- 致命盲区
- 个性化提升路径
- 咨询顾问 CTA
- 登录解锁完整报告 CTA

### 结果页绝对不能出现

- 销售标签
- `P0` / `P1` / `P2`
- 高意向
- 高风险用户
- 直播培育
- 跟进优先级
- 沟通策略
- 开场白参考
- 核心钩子
- 注意避免
- 内部备注
- 运营或销售判断语义

### 分享报告页绝对不能出现

- 任何销售分类或优先级标签
- lead 处理状态
- 后台管理语义
- 客户被打标签、被排序、被分层的表达

### 如果仍然出现上述内容

说明线上运行代码和本地仓库仍有漂移，或者仍有未收回的客户侧渲染逻辑。

处理顺序：

1. 记录具体页面 URL
2. 截图保留
3. 确认是结果页还是分享页
4. 回仓库定位对应渲染模块
5. 直接移除或替换为客户语义模块

## 五、建议截图验收点

建议至少截图保存以下页面：

- 首页
- 结果页上半屏
- 结果页下半屏
- 分享报告页
- `/admin/chat`
- `/lead/mobile`
- `/lead/admin`

其中结果页和分享页截图要特别确认没有销售语义模块。

## 七、当前优先级建议

推荐每次继续做功能前，先过这三个问题：

1. 这个信息是客户需要的，还是销售需要的
2. 这个信息会不会让客户感到被分类、被判断、被销售
3. 这个信息是否应该只存在于 webhook、后台或 lead 页面

只要答案偏向销售判断，就不要放进客户页。
